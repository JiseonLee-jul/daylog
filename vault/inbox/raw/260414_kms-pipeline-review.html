KMS 파이프라인 설계 리뷰

원본 설계에 대한 수정 사항 및 논의점 정리
작성일: 2026-04-14 출처: 02_MASTER-PLAN.md → pipeline-full-design.md 목적: 팀 논의 자료

TL;DR

A (원본 설계): Stage 1이 접수 + 분류(6축) + 라우팅(4경로) + 분해를 모두 담당. Stage 2가 depth로 경로 A/B 분기. Stage 3가 4개 층위(3-a/b/c/d)로 세분화.

B (재구성): Connector → Ingest → Parse → Index로 책임을 평면 분리. 4경로·4 Tier·depth별 청킹 차등 제거. 검증을 그래프 등록 직전으로 당김. critical을 requires_review로 분리.

논의 필요: (1) 처리 원자 단위와 저장 경로 구조, (2) SSOT 원칙 하에서 문서 버전 관리.

§2핵심 비교표

한 줄 요약 — A는 "많은 개념을 한 Stage에 담아 분기로 해결", B는 "각 Stage가 하나의 책임만 지고 분기는 파서 내부 또는 파이프라인 밖으로 위임".

아래 표는 A와 B를 10개 축으로 비교한 것이다. 축 순서는 파이프라인 흐름을 따라간다 — Stage 단위의 책임 구분(1~3행) → 경로·Tier 분기 축소(4~6행) → 메타데이터/검증/배치 재배치(7~10행). 각 축에서 A가 떠안던 복잡성을 B가 어디로 옮겼는지 주목하면 된다.

축 A · 원본 설계 B · 재구성
Stage 책임 한 Stage에 여러 역할이 뭉쳐 있음.
  • Stage 1 = 접수 + 6축 분류 + 4경로 라우팅 + 분해
  • Stage 3 안에 배치 작업·사람 승인까지 포함
  • 변경 영향 범위가 크고 재시도 경계가 불명확
 각 Stage가 하나의 역할만 담당.
  • Connector → Ingest → Parse → Index 평면 분리
  • Stage 경계 = 트랜잭션·재시도 경계
  • 배치·사람 승인은 파이프라인 밖으로 이동
입력 분해 KMS가 소스 시스템의 의미를 알아야 함.
  • "메시지 + URL", "이메일 + 첨부" 분해가 Stage 1 안쪽
  • 소스 시스템이 늘어날수록 Stage 1이 비대해짐
 Connector(KMS 밖)가 분해를 책임.
  • 커넥터가 개별 파일 단위로 kms_ingest() N회 호출
  • KMS는 parent_id만 받음 — "왜 같이 왔는지"는 모름
경로 분기 source_type × doc_type으로 4경로 라우팅.
  • full / lightweight / fast / signal
  • 경로별로 동기·비동기, 파싱 깊이, 청킹 방식이 모두 상이
 외부 경로 분기 없음. 동일 루트 + 내부 분기만.
  • 입력 형태 무관, 산출물은 동일한 마크다운
  • 내부에서 수치 기반 true/false 분기
  • signal은 파이프라인 밖 이벤트 핸들러로
파싱 Tier 4 Tier 체계를 파서 외부에서 제어.
  • Tier 2 기본, 승격 축 4개: 텍스트 레이어 / density / scale / depth
  • Tier 결정이 파서 밖 메타에 의존
 Tier 개념 제거. 파서가 문서를 보고 직접 판단.
  • 판단 축: 확장자 / 표 유무 / 다단 레이아웃
  • 파싱 품질 수치로 내부 분기 (OCR 필요 여부 등)
청킹 결정 축 analytical_depth로 청킹 형태가 결정됨.
  • reference = flat / standard 이상 = parent-child
  • "문서 중요도가 청킹을 결정한다"는 검증되지 않은 가설
 문서의 모양으로 결정. depth와 독립.
  • 축: 타입 + 구조(heading 수) + 크기(token_count)
  • 300p 표 섞인 reference, 3줄 standard 공존 가능
analytical_depth 값 "분석 깊이"와 "사람 검증"이 한 축에 섞여 있음.
  • reference / standard / deep / critical 4단계
  • critical의 실질적 의미 = 사람 검증 필요
 두 축으로 분리해 모든 조합 표현 가능.
  • depth = reference / standard / deep (3단계)
  • requires_review: bool 별도 필드
6축 프로파일 6축 전부를 문서의 고정 속성으로 저장.
  • volatility / density / scale / lifespan / provenance / analytical_depth
  • frontmatter에도 모두 포함
 "상태가 아닌 신호"는 메타가 아니라 트리거로 재배치.
  • volatility → 이벤트 트리거
  • density / scale → 파서 내부 지표
  • lifespan / provenance → Agent 컨텍스트 용도만 유지 검토
검증 시점 그래프 등록 이후 검증 (Stage 3-a).
  • 깨진 데이터가 일단 그래프에 올라간 뒤 검출
  • per-doc 검증과 cross-doc 검증이 같은 시점에 섞임
 시점을 이원화. 성격별로 다른 단계에서 검증.
  • 등록 직전 동기 검증 (per-doc, 스키마·참조·ACL)
  • 등록 이후 배치 검증 (cross-doc, 유사 entity·ACL drift)
합성·배치 파이프라인 내부에 배치 작업이 포함.
  • Stage 3-c = 합성(synthesis), 3-d = 사람 승인
  • "파일 처리"와 "주기적 작업"이 같은 파이프라인에 혼재
 파이프라인 밖 독립 계층으로 분리.
  • 배치 스케줄러: synthesis, consolidation, stale 감지
  • 이벤트 핸들러: 사람 검증, depth 승격
  • 파이프라인은 "문서 1건 처리"만 책임
frontmatter 6축 + 거버넌스 + 요약을 모두 포함.
  • Agent가 필요한 것과 필요 없는 것이 섞임
 Agent 관점으로 범위 축소.
  • 포함: Agent 컨텍스트 + 감사·권한 정보
  • 내부 제어용 메타는 source_record(DB)에만

§3수정 사항 (8건)

각 항목은 핵심 주장 → 배경 → Before/After → 근거 순서로 정리했다. 먼저 굵은 박스의 한 줄만 읽어도 무엇이 바뀌었는지 파악할 수 있다.

01Stage별 책임 분리 (Connector → Ingest → Parse → Index)

한 Stage에 여러 역할이 뭉친 구조를 해체하고, 각 Stage가 하나의 역할만 담당하도록 분리한다. 그래프 등록까지가 파이프라인의 한 단위이고, 배치·사람 승인은 그 밖의 별도 계층으로 뺀다.

배경: A에서는 Stage 1이 접수·분류·라우팅·분해를 모두 떠안고, Stage 3 안에 synthesis 배치 작업과 사람 승인까지 들어가 있다. 이 때문에 "파일 하나를 처리하는 단위"와 "시스템 전체의 주기적 관리 작업"이 같은 파이프라인에 섞여 변경 영향 범위가 커지고 재시도·트랜잭션 경계가 흐려진다.

A · 원본 Stage 1이 접수 + 6축 분류 + 4경로 라우팅 + 복합 입력 분해까지 담당. Stage 3-c에는 배치 작업(synthesis), Stage 3-d에는 사람 승인까지 포함.
B · 재구성 Connector = 복합 입력 분해 (KMS 밖)
Ingest = 파일 수집·저장만 (내용을 읽지 않음)
Parse = 마크다운 변환 + 청킹
Index = 검증 + 임베딩 + 그래프 등록
배치/사람 승인 = 파이프라인 밖 시스템

근거: Beauchemin/Kleppmann의 관심사 분리 원칙. "GraphDB에 등록하는 순간까지"가 파일 하나에 대한 파이프라인의 완결 단위가 되고, 이후의 synthesis·stale 감지·사람 검증은 서로 다른 주기로 움직이는 독립 시스템이므로 분리하는 것이 자연스럽다.

02Stage 1의 4 Path 분기 제거

"파일 형태가 다르면 경로도 달라야 한다"는 전제를 버린다. 입력이 무엇이든 산출물은 동일한 마크다운이라는 점에서 하나의 루트로 통일하고, 동기/비동기 차이는 signal처럼 성격이 다른 신호만 별도 이벤트 핸들러로 뺀다.

배경: A는 source_type + doc_type 조합으로 full / lightweight / fast / signal 4경로를 분기하고 경로별로 처리 방식을 다르게 했다. 하지만 full~fast 세 경로는 결국 "원본을 마크다운으로 바꿔서 인덱싱한다"는 점에서 본질적으로 같은 작업이다. 분기를 외부에 둘 필요 없이 파서 내부에서 확장자·형식별로 처리하면 된다.

A · 원본 source_type × doc_type → 4경로 라우팅. 경로별로 동기/비동기, 파싱 깊이, 청킹 방식이 모두 다르게 설정됨.
B · 재구성 외부 경로 분기 없음. 동일한 파서 루트 위에서 내부의 수치 기반 true/false 분기(텍스트 레이어 유무, 표 유무 등)만으로 처리. signal은 파이프라인 밖 이벤트 핸들러로 분리.

근거: "One axis, one knob" (Tunkelang) — 라우팅 축은 3개 이하로 유지해야 유지보수가 가능하다. full~fast의 차이는 파서 내부의 기술적 선택이지, 파이프라인 수준의 경로가 아니다.

034 Tier 제거

파싱의 목적은 "원본을 얼마나 그대로 잘 베껴오느냐"다. 이는 문서의 중요도·크기·깊이가 아니라 실제 구성 요소(표, 다단 레이아웃 등)가 결정한다. Tier라는 외부 제어축을 제거하고, 파서가 문서를 직접 보고 판단하게 둔다.

배경: A의 4 Tier는 full path 전용으로, Tier 2를 기본으로 두고 텍스트 레이어 유무·density·scale·analytical_depth 네 축으로 Tier 승격을 결정했다. 하지만 파싱 품질은 "이 문서에 표가 있는가, 다단 레이아웃인가, OCR이 필요한가" 같은 문서 구조적 사실로 결정돼야 하며, density/depth 같은 추상적 속성으로 Tier를 외부에서 정하는 것은 제어 위치가 잘못됐다.

A · 원본 Tier 2 기본값 + 4개 축(텍스트 레이어 / density / scale / analytical_depth) 기반 Tier 승격 체계. Tier 선택이 파서 밖에서 결정됨.
B · 재구성 Tier 개념 제거. 확장자·표·레이아웃 같은 구조적 사실로 파서 내부에서 자체 판단. 중간중간 파싱 품질 수치 기반 true/false 분기(OCR 필요 여부 등).

근거: 파싱 품질의 3차원(Completeness / Fidelity / Structural Integrity)은 문서 구조로 결정되는 속성이고, density·scale·depth 같은 메타와는 다른 층이다. 파서 내부로 캡슐화해야 "한 문서에 표가 나오면 표 파이프라인을 태우는" 것 같은 자연스러운 처리가 가능하다.

04volatility / density / scale / lifespan / provenance 메타데이터 재배치

6축 프로파일 중 대부분은 "문서의 고정 속성"이 아니라 "시스템을 움직이는 신호"다. 신호는 메타가 아니라 트리거/파서 내부 지표로 옮기는 것이 맞다. 문서에 남겨야 할 것은 Agent 검색에 필요한 최소 컨텍스트뿐이다.

배경: A는 6축 모두를 문서의 고정 속성으로 저장하고 frontmatter에도 전부 포함했다. 그러나 실제로 이 축들이 하던 역할을 다시 살펴보면, 상당수는 "이 문서가 어떻다"라는 상태보다 "언제 무엇을 트리거하라"는 신호에 가깝다. 신호를 상태로 저장하면 동기화 문제가 생긴다.

  • volatility — signal 경로 힌트 + 재처리 트리거로 사용됨. 둘 다 파이프라인 밖 이벤트 트리거로 옮기면 충분. 문서에 저장할 메타가 아님.
  • density — Tier 결정과 L0 길이 조정에 사용됨. Tier 개념 자체가 사라졌고, L0 길이는 analytical_depth로 통일. 남은 용도는 파서 내부 지표 뿐.
  • scale — micro일 때 fast path 분기, L0/L1 길이, parent-child 청킹 채택에 사용됨. 경로는 사라졌고 L0/L1은 depth로 통일. 청킹은 depth와 독립으로 분리되어 scale은 청킹 로직 내부 지표가 됨.
  • lifespan / provenance — 배치 스케줄러·이벤트 트리거 + Agent의 검색 컨텍스트 제공에 사용됨. Agent 컨텍스트 용도로만 유지 검토.

부연: density는 source_type + doc_type에 사실상 종속된 파생값이므로 별도 속성으로 저장할 이유가 더욱 약하다.

05analytical_depth 재정립

"분석 깊이"와 "사람 검증 필요성"은 서로 다른 축이다. A는 이 둘을 하나의 depth 값에 섞어 critical을 만들었지만, B는 두 축을 분리해 모든 조합을 표현할 수 있게 한다.

배경: A의 critical은 사실상 "사람 검증이 필요한 문서"를 뜻한다. 그런데 이는 분석 깊이와는 직교하는 개념이다. 예를 들어 파싱 품질이 낮은 reference 문서나 검색에서 부정 피드백을 받은 standard 문서도 분석 깊이와 무관하게 사람 리뷰가 필요할 수 있다. 두 축을 섞어놓으면 이런 조합을 표현할 수 없다.

A · 원본 reference / standard / deep / critical 4단계.
critical과 deep의 실질적 차이 = 사람 검증 여부.
B · 재구성 depth = reference / standard / deep 3단계
+ requires_review: bool 별도 필드
→ 두 축의 모든 조합 표현 가능.

확장성: requires_review를 별도 축으로 두면 검색 횟수, 부정 피드백, 파싱 품질 점수 등을 트리거로 해서 이벤트 핸들러가 requires_review를 동적으로 true로 전환하는 확장이 자연스럽게 가능해진다.

06frontmatter · source_record 스키마 정리

frontmatter에는 Agent가 실제로 사용하는 컨텍스트 + 감사·권한 정보만 남긴다. 6축 프로파일처럼 내부 제어용 필드는 frontmatter에서 빼고 DB 테이블에만 둔다.

배경: A의 frontmatter에는 6축 프로파일과 거버넌스 필드가 모두 들어가 있었다. 그러나 frontmatter는 Agent가 문서를 이해하기 위해 읽는 영역이고, 내부 처리용 메타는 Agent에게 보이지 않아도 된다. 관심사를 분리해서 Agent 관점의 frontmatter와 시스템 관점의 source_record를 나눠야 한다.

source_record는 19개 필드로 정리: 식별(5) + 무결성(2) + 감사(4) + 권한(3) + 처리 제어(2) + 상태(2). frontmatter는 14개 필드로 축소: 식별(2) + 출처(3) + 요약(2) + 지식 신호(2) + 콘텐츠 메타(1) + 상태(3).

07청킹 결정 축 변경

청킹을 결정하는 축은 문서의 중요도(depth)가 아니라 문서의 모양(타입·구조·크기)이다. 청킹은 Agent가 받는 컨텍스트의 기본 단위이고, depth는 지식 구조화의 수준이라서 서로 다른 층이다.

배경: A는 "reference = flat chunk, standard 이상 = parent-child"로 depth가 청킹 형태까지 결정하게 설계했다. 하지만 이 가설은 현실과 충돌한다 — 300페이지 표가 섞인 reference 문서, 3줄짜리 standard 문서가 공존 가능한데 depth만으로 청킹을 결정하면 둘 다 잘못된 형태로 쪼개진다. 리서치 결과 학술 논문·프로덕션 시스템·LightRAG 모두 문서 타입·구조·크기로 청킹을 결정하며, depth별 청킹 차등은 어디에서도 검증되지 않았다.

A · 원본 analytical_depth 기반 차등.
reference = flat
standard 이상 = parent-child
B · 재구성 문서 타입 + 구조(heading 개수) + 크기(token_count) 3개 축으로 결정. depth와 완전히 독립.
· 타입: 코드/표/채팅/산문
· 구조: heading 충분/부족
· 크기: <2K flat, ≥2K + 다단 → parent-child

08검증 시점 변경 (이원화)

"깨진 데이터를 그래프에 올리지 않는" 검증은 인덱싱 전에, "cross-doc으로 중복·모순을 찾는" 검증은 배치로. 한 종류의 검증을 한 시점에 몰아넣지 말고, 성격에 맞게 이원화한다.

배경: A는 Stage 3-a에서 그래프 등록 이후 검증을 수행한다. 이 순서는 깨진 데이터가 일단 그래프에 올라간 뒤 검출된다는 뜻이다. 반면 문서 형식·참조 무결성 같은 per-doc 검증은 본질적으로 동기적이고 선제적으로 걸러낼 수 있으며, 유사 entity 병합·ACL drift 같은 cross-doc 검증은 여러 문서를 모아서 주기적으로 봐야 하는 성격이라 시점 자체가 다르다.

A · 원본 Stage 3-a에서 그래프 등록 이후 검증. per-doc 검증과 cross-doc 검증이 같은 시점에 섞여 있음.
B · 재구성 등록 전 동기 검증 (per-doc: 스키마, 참조 무결성, ACL 매칭, 링크 유효성) + 파이프라인 밖 배치 검증 (cross-doc: 유사 entity 병합, ACL drift, 고아 entity). 두 검증의 주기와 범위가 다르므로 분리.

원칙: defense in depth — 같은 문서에 대해 Stage 2(구조)와 Stage 3(의미) 두 번 검증하는 것은 낭비가 아니라 안전장치다. 반면 여러 문서를 아우르는 검증을 파이프라인 안에 넣으면 단일 문서의 지연·실패가 전체를 막게 된다.

§4논의 필요 사항 (2건)

① 지식 입력 vs 처리의 원자 단위 · 저장 경로 구조

Q. 입력 단위(쓰레드 + 첨부파일 N개)와 처리 단위(파일 1개) 사이의 관계를 어떻게 표현할까?

  • 저장 경로 구조 변경? /raw/{source_id}/original.* → /raw/{ingest_id}/original.*
  • 하나의 원자적 입력에 여러 데이터 소스가 들어올 수 있음. 그 간의 관계를 그래프 노드 상에 명시하는 것이 맞을까?
  • 문서 간의 계층 구조(parent_id, has_attachment)를 유지하는 것이 맞나?
옵션 A · 그룹 유지ingest_id + parent_id + has_attachment 엣지로 관계를 그래프에 명시적으로 표현
옵션 B · 평면화파일이 유일한 원자 단위. 그룹 개념은 저장 경로에만 반영하고 그래프에는 표현 안 함

② SSOT 원칙 하에서의 문서 버저닝

Q. 외부 원본이 SSOT, /raw/·/transformed/는 원본 스냅샷으로 간주. 그렇다면 수정이 잦은 원본은?

  • 예: Google Drive의 한 문서를 여러 명이 계속 수정. 하나의 문서에 여러 스냅샷이 생김.
  • A 스냅샷과 B 스냅샷을 아예 다른 대상으로 볼 것인가, 동일 대상의 버전으로 볼 것인가?
옵션 A · 독립 문서각 스냅샷을 독립 source_record로. 단순하지만 중복·탐색 비용 발생
옵션 B · 버저닝canonical_id + version 체계. 관계는 명확하나 변경 감지·재처리 파이프라인 필요

연관 주제: Federated Origin + Cached Derivatives 원칙 / 원본 변경 감지 이벤트 / 문서 재처리 범위 (Parse부터? Index만?)
