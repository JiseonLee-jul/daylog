KMS · Design Summary

Evaluation Harness

Author 이지선  ·  Date 2026-04-17  ·  Draft 팀 공유용
01 목적
KMS는 단번의 설계로 완성되지 않는다. 설계하고, 실 데이터에 부딪히고, 문제를 발견하고, 컴포넌트를 뜯어 고치는 과정이 반복될 수밖에 없다. 따라서 핵심은 문제를 빠르게 판단하고, 원인을 분석하고, 개선을 검증하는 루프를 체계적으로 갖추는 것이며, 그 중심에 Evaluation Harness가 있다.
평가 대상
파이프라인 각 단계(Ingest → Parse → Index) + Knowledge Graph 품질
요구사항
  • 독립 컴포넌트가 교체되어도 평가 코어는 무수정. Evaluator만 유일한 수정 지점.
  • 분산 로컬과 Celery 분산 환경 모두에서 일관 동작.
  • 확장 개발 도구에서 출발, 운영 관측·감사 대시보드의 기반으로 확장.

03 왜 빠른 루프인가
컴포넌트 격리
컴포넌트를 바꿔도 평가는 안 깨진다
Component Base Class
  __call__(): 계측+에러처리 (고정)
  _run():     순수 로직 (교체 가능)
       │ 상속
  ┌────┼────┐
  Parser  Chunker  Embedder …
  (교체해도 평가 무수정)
모든 컴포넌트가 같은 Base Class를 상속 → 평가가 보는 인터페이스가 불변
컴포넌트 교체? _run()만 새로 구현. 평가 코드 무수정.
확장성
평가 항목을 YAML 한 장으로 추가한다
YAML: metric: chrf_plus
       │ 문자열
Registry: {"chrf_plus": ChrFPlus, …}
       │ 클래스 바인딩
ChrFPlus.compute() → 0.91
Registry가 이름으로 클래스를 연결 → YAML에 이름만 쓰면 됨
새 메트릭? metrics/ 파일 + YAML 한 줄. 코어 무수정.
새 Evaluator? evaluators/ 클래스 + YAML 한 장.
벤더 독립
대시보드를 자유롭게 바꿀 수 있다
L1 Pipeline  → OTel Traces  ─┐
                              ├→ OTLP → ?
L2 Evaluator → OTel Metrics ─┘
                                   │
                            ┌──────┼──────┐
                           CLI  Langfuse  자체
                          (지금)  (다음)  (나중)
OTLP 공용 규격으로 보내기만 → 받는 쪽 자유롭게 교체
평가 워커는 백엔드에서 데이터를 꺼내오지 않음 → 종속 자체가 없음
04 추후 과제
평가 체계 상세화
  • YAML 필드 명세 (depth별 평가 분기, 가중치 정의 등)
  • 그래프 평가 구체화 (golden 형태, entity 매칭 전략)
  • 회귀 판정 상세 정책 (baseline 저장, 임계값, 통계 검정)
운영 연동
  • CI/CD 도구 선정 및 연동
  • Golden 관리 체계 (작성→리뷰→승인 워크플로)
  • 통합 인터페이스 (대시보드 UI, Claude Code 연동 등)
파이프라인 연동
  • 실행 메타(소요시간, 토큰 수) 저장 경로 확정
  • 병렬 실행 구간 실패 정책
