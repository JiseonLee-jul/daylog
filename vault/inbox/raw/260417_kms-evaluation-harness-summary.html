KMS · Design Summary

Evaluation Harness

Author 이지선  ·  Date 2026-04-17  ·  Draft 팀 공유용
01 목적
KMS는 단번의 설계로 완성되지 않는다. 설계하고, 실 데이터에 부딪히고, 문제를 발견하고, 컴포넌트를 뜯어 고치는 과정이 반복될 수밖에 없다. 따라서 핵심은 문제를 빠르게 판단하고, 원인을 분석하고, 개선을 검증하는 루프를 체계적으로 갖추는 것이며, 그 중심에 Evaluation Harness가 있다.
R1 컴포넌트 독립
컴포넌트가 교체되어도 평가 코어는 무수정. Evaluator만 유일한 수정 지점.
R2 파이프라인 + 그래프
파이프라인 각 단계와 Knowledge Graph 둘 다 평가.
R3 인터페이스 변화 내성
필드는 바뀌어도 흐름 구조에만 결합. 변경 영향 최소화.
R4 분산 실행 대응
로컬과 Celery 분산 환경 모두에서 일관 동작.
R5 장기 확장
개발 도구에서 출발, 운영 관측·감사 대시보드의 기반으로 확장.
R6 회귀 판정 + 재현성
결정적 메트릭은 즉시 차단, 비결정적 메트릭은 통계적 판정.

03 왜 빠른 루프인가
컴포넌트 격리
컴포넌트를 바꿔도
평가는 안 깨진다
Parser를 Docling에서 Marker로 교체해도 평가 코드 무수정
평가 코어(Metric, Group, Registry)는 컴포넌트 변경과 완전 격리
빠르게 교체 실험 → 즉시 결과 비교 가능
확장성
평가 항목을 YAML 한 장으로 추가한다
새 메트릭이 필요하면 파일 하나 + YAML 한 줄
코어 코드를 건드리지 않음
평가 기준을 빠르게 확장 가능
벤더 독립
대시보드를 자유롭게 바꿀 수 있다
지금은 CLI + 파일 결과, 나중엔 Langfuse, 그 다음엔 자체 대시보드
OTel(OTLP)로 방출하므로 수신 도구에 종속 없음
도구가 바뀌어도 파이프라인/평가 코드 무수정
04 개발 가이드
상황 하는 일
파이프라인 컴포넌트 추가/교체 Component 상속, _run()만 구현. OTel은 자동
평가 항목 추가 evaluators/ 클래스 + configs/ YAML 한 장
새 메트릭 추가 metrics/ 파일 + YAML에 한 줄
평가 실행 python -m kms.eval.run --evaluator parser_pdf
결과 확인 runs/{run_id}/results.json, git으로 이력 관리
