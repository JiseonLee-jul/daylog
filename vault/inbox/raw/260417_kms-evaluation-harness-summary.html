측정 루프를 설계한다 —
Evaluation Harness

Date 2026-04-17  ·  Author KMS Team  ·  Status Draft / 팀 공유용
01 목적
KMS는 단번의 설계로 완성되지 않는다. 컴포넌트를 교체·개선하며 점진적으로 발전시키는 체계가 필요하다. 그 중심에 Evaluation Harness — 빠르게 측정하고, 해석하고, 개선을 검증하는 루프 — 가 있다.
02 아키텍처
L3 Continuous Evaluation — 언제 평가하는가
Local (dev) CI (PR gate) Nightly (full suite) Shadow (prod traffic)
↓
L2 Evaluation Harness — 무엇을 측정하는가
Evaluator
평가 단위. ArtifactLoader + GoldenLoader로 데이터를 로드하고 실행 결과를 Metric에 위임
Metric
개별 측정 함수. plugin 방식으로 Registry에 등록. @register_metric
Group + Comparator
여러 Evaluator 집계. 이전 run 대비 회귀/개선을 판정. OTel Metrics로 방출
L1 + L2 횡단 인프라
OTel — Traces (L1) + Metrics (L2) Shared Storage — S3 / DB (artifact + golden)
↓
L1 Pipeline — 비즈니스 로직
Component Base Class
모든 컴포넌트의 루트. OTel 계측·에러처리 내장. 서브클래스는 _run()만 구현
LLMComponent
LLM 호출 전용 서브클래스. gen_ai.* 속성 자동 기록
Pipeline Workers
Celery 기반 비동기 실행. 완료 후 평가 워커를 트리거
L1 전용 인프라
Celery Workers Trace Context (propagation)
데이터 흐름 — L1 → L2
L1 Component __call__ 실행 — 두 경로로 분기:
① artifact + 실행 메타 → S3/DB — 평가 워커가 읽는 유일한 소스
② OTel Traces → OTLP → 대시보드 — 시각화 전용, 꺼내오지 않음
L2 파이프라인 워커 완료 후 비동기로 평가 워커 트리거
L2 ArtifactLoader + GoldenLoader → Evaluator → Group → Comparator → OTel Metrics 방출
OTel Langfuse / MLflow / Grafana 등 어떤 백엔드든 교체 가능 — 벤더 종속 없음
03 핵심 개념
Component Base Class
  • 모든 파이프라인 컴포넌트가 상속
  • __call__이 OTel 계측 + 에러처리를 은닉
  • 서브클래스는 _run()만 구현 — OTel 코드 0줄
  • LLM 호출 컴포넌트는 LLMComponent 상속, gen_ai.* 자동 기록
Evaluator / Metric / Group
  • EleutherAI lm-evaluation-harness 패턴 차용
  • 평가 정의 = YAML(데이터) + Metric(plugin) + Group(집계)
  • @register_evaluator, @register_metric으로 동적 등록
  • 새 메트릭 추가 = metrics/ 파일 + YAML 한 줄 — 코어 무수정
OTel (OpenTelemetry)
  • 평가의 인프라 계층 — 관측 도구가 아닌 관측의 공용 규격
  • L1은 Traces를, L2는 Metrics를 OTLP로 방출
  • 방출 전용 — 백엔드에서 꺼내오지 않음 → 벤더 종속 없음
  • Langfuse, MLflow, Grafana 등 어떤 대시보드든 교체 가능
04 개발 가이드
상황 하는 일
파이프라인 컴포넌트 추가/교체 Component 상속, _run()만 구현. OTel은 Base Class가 자동 처리
평가 항목 추가 evaluators/에 클래스 + configs/evaluators/에 YAML 한 장
새 메트릭 추가 metrics/에 파일 + YAML metric_list에 한 줄 추가
평가 실행 python -m kms.eval.run --evaluator parser_pdf
결과 확인 runs/{run_id}/results.json → git으로 이력 관리
