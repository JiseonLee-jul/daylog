# Shopify의 데이터 구조화 여정: One-Shot LLM에서 DSPy 기반 에이전틱 아키텍처로

> 출처: https://news.hada.io/topic?id=28022

## 문제 배경

Shopify에는 수백만 개의 상점이 있는데, 각 상점마다 HTML 구조, 언어, 정책 표기 방식이 전부 다르다. 그래서 "이 상점이 휴대폰을 파는가?", "반품 정책은 무엇인가?" 같은 단순한 질문에도 표준화된 답변을 자동으로 뽑아내기가 어려웠다. 이 비정형 데이터를 구조화하는 게 핵심 과제.

## 솔루션 진화 단계

### 1단계: One-Shot LLM

상점 페이지 텍스트를 통째로 GPT-4/5에 넘기고 "이 상점 정보를 스키마에 맞게 뽑아줘"라고 요청하는 방식. 간단하지만, 컨텍스트 윈도우 제한에 걸리기 쉽고, 프롬프트가 조금만 바뀌어도 결과가 흔들렸다. 상점 수가 많아질수록 비용이 기하급수적으로 증가하는 것도 문제.

### 2단계: 에이전틱 접근 + DSPy

LLM이 단순히 텍스트를 받아서 답하는 게 아니라, 직접 브라우징하고 조사하는 도구를 사용할 수 있게 했다(ReAct 에이전트 구조). 프롬프트를 사람이 수동으로 다듬는 대신, DSPy 옵티마이저가 프로그래밍 방식으로 프롬프트를 자동 최적화한다는 점이 핵심 전환.

### 3단계: 전문화된 서브 에이전트

하나의 에이전트가 모든 걸 처리하니 서로 간섭이 생기는 문제가 있었다. 이를 역할별로 쪼갬:
- **Fraud Agent**: 외부 리뷰 사이트를 검색해서 사기 여부 판별
- **Profile Agent**: 상점 내부 페이지를 파싱해서 정책/프로필 정보 추출

각 에이전트를 독립적으로 최적화할 수 있어서 성능 향상이 훨씬 쉬워졌다.

## 기술적 해결책: ShopNap 스냅샷 서비스

에이전트 성능을 평가하려면 "정답 라벨"이 필요한데, 실시간 크롤링을 하면 웹사이트가 시시각각 바뀌어서 라벨과 실제 데이터가 어긋났다. 이를 해결하기 위해 라벨링 시점의 상점 상태를 통째로 스냅샷으로 고정해두는 서비스를 만들었다. 평가의 재현성을 보장하는 장치.

## 인프라 아키텍처 (3계층)

- **Batch Layer (Flink)**: 일일 15만 개 이상 상점을 배치 처리
- **Agent Layer (Kubernetes)**: CPU 클러스터에서 에이전트 로직(도구 호출, 판단 흐름) 수행
- **LLM Layer (GPU Cluster)**: vLLM으로 자체 호스팅한 Qwen 모델이 추론 담당

GPU 비용이 비싼 LLM 추론과, CPU로 충분한 에이전트 로직을 분리해서 비용 효율을 높인 구조.

## 성능 비교

| 항목 | One-Shot (GPT-5) | Agentic + DSPy + Qwen |
|------|------------------|----------------------|
| 비용 | 기준점 | 1/75 수준 |
| 품질 | 기준점 | 약 2배 향상 |
| 커버리지 | 일부 (비용 제약) | 전체 상점 |
| 확장성 | 전체 재검증 필요 | 서브 에이전트 추가로 간편 |

## 핵심 교훈

1. **전문화의 중요성**: 복잡한 작업일수록 단일 에이전트보다 관심사를 분리한 서브 에이전트 구조가 유리하다.
2. **아키텍처 우선**: 프롬프트 문구를 하나하나 손보는 것보다, 시스템 아키텍처를 잘 설계하고 최적화를 자동화하는 편이 지속 가능하다.
3. **소형 모델의 승리**: 범용 대형 모델(GPT-5)보다 도메인에 맞게 최적화한 중소형 모델(Qwen)이 가성비와 성능 모두 앞섰다.
