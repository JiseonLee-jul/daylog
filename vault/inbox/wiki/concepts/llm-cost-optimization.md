# LLM Cost Optimization

대규모 LLM 운용의 비용을 줄이기 위한 아키텍처 전략. 단순히 저렴한 모델을 쓰는 게 아니라, **작업 특성에 맞는 모델 선택**, **도메인 특화 소형 모델 활용**, **인프라 계층 분리**, **파이프라인 최적화 자동화** 등의 복합적 접근이 필요하다.

## Key Points

- **소형 특화 모델 > 범용 대형 모델**: 도메인에 맞게 최적화된 중소형 모델이 범용 대형 모델보다 가성비와 정확도 모두 앞설 수 있음
- **인프라 계층 분리**: CPU로 충분한 에이전트 로직과 GPU가 필요한 LLM 추론을 물리적으로 분리해 리소스 낭비 방지
- **자체 호스팅**: vLLM 같은 효율적인 서빙 프레임워크로 자체 호스팅 시 호출당 비용 대폭 절감
- **배치 처리**: 일일 15만+ 아이템 같은 대용량 작업은 Flink 같은 배치 레이어로 처리
- **파이프라인 자동 최적화**: DSPy 같은 프레임워크로 프롬프트 튜닝 비용 제거
- **서브 에이전트 분리**: 전문화된 작은 에이전트 여러 개가 거대 단일 에이전트보다 저렴하고 유지보수 쉬움

## Shopify의 3-Layer 비용 최적화 아키텍처

Shopify는 데이터 구조화 시스템에서 3개 계층으로 비용을 최적화했다:

| 계층 | 기술 | 역할 |
|------|------|------|
| Batch Layer | Flink | 일일 15만+ 상점 배치 처리 |
| Agent Layer | Kubernetes (CPU) | 에이전트 로직, 도구 호출, 판단 흐름 |
| LLM Layer | vLLM (GPU) | 자체 호스팅한 Qwen 모델 추론 |

이 분리의 핵심은 **비싼 LLM 추론(GPU)과 CPU로 충분한 에이전트 로직을 분리**한 것이다. 만약 에이전트 로직까지 GPU 클러스터에서 돌린다면 GPU 사용률이 낮아 비용 효율이 떨어진다.

## 결과

Agentic + DSPy + Qwen 조합은 One-Shot GPT-5 대비:

- **비용**: 1/75 수준
- **품질**: 약 2배 향상
- **커버리지**: 일부(비용 제약) → 전체 상점

이는 "비용 절감과 품질 향상은 반드시 트레이드오프가 아니다"는 반례가 된다. 올바른 아키텍처 선택과 도메인 특화 모델 활용으로 두 가지를 동시에 개선할 수 있다.
