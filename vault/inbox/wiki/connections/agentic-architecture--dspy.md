# Agentic Architecture ↔ DSPy

DSPy는 agentic architecture의 **파이프라인을 선언적으로 구성하고 자동 최적화하는 프레임워크**다. Agentic 시스템은 본질적으로 여러 LLM 호출과 도구 사용으로 구성되어 있어 프롬프트 튜닝의 조합 폭발이 심각한데, DSPy는 이 복잡도를 프로그래밍 패러다임으로 해결한다. 두 개념의 관계는 "구조와 최적화"의 관계다.

## From Agentic Architecture perspective

Agentic 시스템의 가장 큰 운영 부담은 **프롬프트 유지보수**다. 각 에이전트의 시스템 프롬프트, 도구 설명, 응답 파싱 규칙이 모두 수동 튜닝의 대상이 되면, 에이전트가 늘어날수록 유지보수 비용이 기하급수적으로 증가한다. DSPy는 이 문제를 해결하는 "OS 레이어"다:

- **모듈화**: 에이전트 파이프라인을 재사용 가능한 모듈(Signature, Module)로 분리
- **자동 최적화**: 옵티마이저가 학습 데이터와 메트릭을 기반으로 프롬프트를 자동 생성
- **독립 최적화**: 전문화된 서브 에이전트를 각자 독립적으로 튜닝 가능
- **스케일링 친화**: 에이전트 수가 늘어도 선언형 구조 덕에 관리 부담이 선형 증가

## From DSPy perspective

DSPy는 단순 LLM 호출 래퍼가 아니라 **agentic system을 일급 시민으로 다루는** 프레임워크다. ReAct 패턴, Chain-of-Thought, 도구 사용 같은 agentic 기본 요소가 DSPy의 핵심 추상화에 녹아 있다. DSPy가 진정한 가치를 발휘하는 곳은:

- **다단계 파이프라인**: 단일 호출이 아닌 여러 LLM 호출이 연결된 복잡한 흐름
- **도구 사용 에이전트**: ReAct 같은 관찰-추론-행동 루프가 필요한 경우
- **성능/비용 트레이드오프 관리**: 어떤 단계에 어떤 모델을 쓸지 최적화
- **서브 에이전트 분리**: 관심사가 분리된 여러 에이전트를 선언적으로 구성

## Shopify 사례의 증명

Shopify는 이 관계를 실제 프로덕션에서 입증했다:

1. **One-Shot LLM**: 거대 단일 호출로 모든 작업을 처리 → 한계 명확
2. **Agentic + DSPy**: ReAct 에이전트가 도구 사용, DSPy가 프롬프트 자동 최적화
3. **전문화된 서브 에이전트**: Fraud/Profile 에이전트로 관심사 분리, 각각 DSPy로 독립 최적화

결과: 비용 1/75, 품질 2배. 이 수치는 "agentic architecture + DSPy" 조합이 단순한 이론적 개선이 아니라 **프로덕션에서 검증된 전략**임을 보여준다. 핵심 교훈 "프롬프트 문구를 하나하나 손보는 것보다, 시스템 아키텍처를 잘 설계하고 최적화를 자동화하는 편이 지속 가능하다"는 두 개념의 관계를 한 문장으로 요약한다.
