## Part 1. 뤼튼의 AX 사업 현황

### 1-1. 뤼튼은 누구인가

뤼튼테크놀로지스는 "Bring AGI Close to People"을 비전으로, 1년 7개월 만에 MAU 700만을 달성한 국내 1위 AI 서비스 기업이다. 2025년 3월 Series B 830억원을 유치했으며, 2024년 9월 B2B 전담 조직인 **wrtn.ax**를 설립해 삼성, CJ, KT, LG, 기아 등 대기업과 정부기관을 대상으로 AI 전환(AX) 사업을 본격화하고 있다.

### 1-2. AX에 대한 뤼튼의 관점

> "AX는 에이전트를 구축하는 프로젝트가 아니라, 생산성 병목을 해결하는 프로젝트다."

뤼튼은 AX를 기술 도입이 아닌 **경영 혁신**으로 정의한다. 기술은 이미 상향평준화되었고, 차이를 만드는 것은 "어떤 병목을 찾아 어떻게 해결했는가"의 실전 경험이라는 것이 핵심 메시지다.

### 1-3. 자사 AX 실증 결과

뤼튼은 자사 4개 부서에 AI 에이전트를 직접 적용하고, 그 성과를 수치로 공개했다.

| 적용 영역 | 역할 | 생산성 증가 | 월간 투입 공수 감소 |
|---------|------|---------|--------------|
| **고객 응대 (CX)** | 고객 문의 응답부터 환불/재배송 처리까지 자동화 | 35% | 73% (1,543분 → 423분) |
| **재무 (Finance)** | 회계, 지출, 급여, 재무제표 작성 자동화 | 21% | 40% (823분 → 496분) |
| **프론트엔드 개발** | 디자인 → 코드 변환, API 연동, 빌드/테스트 자동화 | 15% | 28% (3,500분 → 2,513분) |
| **백엔드 개발** | 자연어 → API/DB 설계, 테스트, 코드 자동 생성 | 28% | 55% (3,000분 → 1,348분) |

### 1-4. 핵심 방법론

#### 3-Layer 프레임워크

뤼튼은 모든 업무를 난이도에 따라 3개 계층으로 분류하고, 아래 계층부터 순차적으로 자동화한다.

| 계층 | 업무 성격 | 예시 |
|------|---------|------|
| **Layer 1** | 반복적이고 규칙이 명확한 업무 | FAQ 응답, 급여 명세서 처리, 빌드 스크립트 작성 |
| **Layer 2** | 데이터 분석과 일차적 판단이 필요한 업무 | 고객별 맞춤 응답, 재무 분석 리포트, 요구사항 분석 |
| **Layer 3** | 최종 의사결정과 외부 시스템 연동이 필요한 업무 | 환불 실행, 재무 예측(FP&A), 코드 품질 분석 후 개선 |

이 프레임워크의 본질은 **도입 순서**다. Layer 1에서 빠르게 성과를 내고, 그 수치를 근거로 다음 계층의 투자를 정당화하는 구조다.

#### 성과 측정 체계

뤼튼은 두 가지 지표로 AX의 효과를 정량화한다.

**투입 공수 측정:** 모든 업무를 최소 단위(건)로 정의하고, 건당 평균 처리 시간을 AI 도입 전후로 비교한다. 예를 들어 고객 응대 부서의 FAQ 처리는 건당 19분에서 3분으로 줄었다(-85%).

**생산성 지수:** 자체 개발한 수학적 모델(AI Productivity Index)로 "인간+AI 협업 시 총 생산성이 얼마나 증가했는가"를 단일 숫자로 산출한다. "35% 생산성 증가"처럼 경영진과 고객사에 직관적으로 전달할 수 있는 형태다.

#### 비용 통제: Model Orchestration

모든 작업에 최고 성능의 AI 모델을 쓰면 비용이 감당이 안 된다. 뤼튼은 자체 LLM 평가 방법론(KLAUD)으로 모델별 강점 영역을 프로파일링한 뒤, 작업 특성에 따라 최적의 모델을 자동 배치한다. 구조화된 패턴 처리는 저비용 ML 모델이, 맥락 이해가 필요한 작업은 LLM이 담당하는 하이브리드 구조로, 품질은 유지하면서 비용을 통제한다.

### 1-5. AX 성공 3요소와 실패 5요소

뤼튼은 다수의 기업 AX 프로젝트 경험을 바탕으로 성공과 실패의 조건을 다음과 같이 정리했다.

**성공 3요소:**

| 요소 | 내용 |
|------|------|
| **생산성 병목 정의** | 해결할 문제가 명확히 정의되어 있는가 |
| **기술력** | Model Orchestration/RAG/MCP 등 사내/외부 기술의 경계가 명확하고 실행 가능한가 |
| **경영진의 의지와 이해** | 실무 현장의 니즈를 이해하고 해결하려는 의지가 있는가 |

**실패 5요소:**

| 요소 | 내용 |
|------|------|
| **Quality** | LLM 성능이 기업 현장이 원하는 수준에 못 미치는 경우 |
| **Price** | AI 도입/활용 비용이 기존 비용을 크게 상회하는 경우 |
| **Data Infra** | 사내 데이터가 디지털화되어 있지 않아 AI가 활용할 수 없는 경우 |
| **UX** | UX 설계 미흡으로 구성원들이 사용하지 않는 경우 |
| **AI Literacy** | 사내 직원들의 AI 이해도/활용도가 떨어지는 경우 |

### 1-6. B2B 사업 모델: AX Triangle Package

뤼튼은 제품 단독 판매가 아닌, 세 가지를 묶은 패키지 형태로 B2B 사업을 전개한다.

| 구성 | 내용 |
|------|------|
| **AI 역량 강화 프로그램** | 고객사 구성원의 AI 이해도를 높여 도입 저항을 줄이는 교육 |
| **AX 컨설팅** | 기업별 상황에 맞는 병목 진단, AX 로드맵 수립, 에이전트 구조 설계 |
| **에이전트 구축** | 로드맵 우선순위에 따른 실제 AI 에이전트 개발 및 배포 |

뤼튼이 발견한 실패들은 공통적으로 **진단과 교육 없이 구축부터 시작**했거나, **구축 역량 없이 계획만 세운** 경우다.

## Part 2. 우리의 벤치마킹 전략

### 2-1. 전략의 프레임

뤼튼이 정의한 성공 3요소와 실패 5요소를 전략의 최상위 프레임으로 삼는다. 성공 3요소는 "확보해야 할 것", 실패 5요소는 "전 과정에서 회피해야 할 가드레일"이다. 이하의 모든 전략(도입 프로세스, 기술력 고도화, 성과 측정, Dogfooding, 패키지 판매)은 이 프레임 안에서 각 요소를 확보하거나 회피하기 위한 구체적 수단이다.

```
┌─ 성공 3요소 (확보해야 할 것) ──────────────────────────────────────┐
│                                                                    │
│   [병렬 트랙 A]                [병렬 트랙 B]                        │
│   요소1. 생산성 병목 정의       요소2. 기술력                        │
│   - Step 1. 적합성 판단        - 필수 기술 체크리스트 + 현황 평가     │
│   - Step 2. 병목 진단          - 항목별 고도화 세부 전략 수립         │
│   - 기준선 측정                - 모델 스펙 정리표 구축                │
│          ↓                            ↓                            │
│          └───────────┬────────────────┘                            │
│                      ↓                                             │
│   Step 3. 3-Layer 분류 (병목 + 기술력이 합쳐져 현실적 로드맵)        │
│                      ↓                                             │
│   Step 4. 구축 + 측정 (Layer 1부터, Dogfooding)                     │
│                      ↓                                             │
│   요소3. 경영진의 의지와 이해                                        │
│   - 성과 측정 체계로 지속적 설득                                      │
│   - Layer 1 Quick Win → 다음 투자 정당화                             │
│                                                                    │
├─ 실패 5요소 (전 과정의 가드레일) ──────────────────────────────────┤
│   Quality     → 모델 스펙 정리표 + 고객사 상황별 모델 추천 역량       │
│   Price       → 비용 제약 선파악 + 모델 최적 조합 + 비용 절감 기술    │
│   Data Infra  → 자사 데이터 정리 과정에서 가이드라인 축적 → 컨설팅화  │
│   UX          → 인터페이스 선택지 제공 + UX 노하우 + 피드백 체계      │
│   AI Literacy → 레벨별 교육 프로그램 + CX Agent로 상시 지원 확장      │
│                                                                    │
├─ 횡단 전략 (여러 요소에 걸쳐 작동) ────────────────────────────────┤
│   Dogfooding  → 전 요소에서 자사 적용 → 경험 축적 → B2B 자산화       │
│   성과 측정   → 요소1(기준선) + 요소3(설득) + Price(정당화)          │
│   패키지 판매 → 전체 경험의 B2B 사업화                               │
└────────────────────────────────────────────────────────────────────┘
```

### 2-2. 성공 요소 1 — 생산성 병목 정의: AX 도입 프로세스

**Step 1~2를 건너뛰고 구축부터 하면 실패한다.** 뤼튼이 관찰한 시장 실패의 공통 원인이다.

#### Step 1. 적합성 판단 — "이 업무에 AI를 적용할 수 있는가?"

모든 업무에 AI를 적용할 수 있는 것은 아니다. 인터뷰로 업무를 수집한 뒤, 다음 네 가지 조건을 모두 충족하는 업무만 AX 대상으로 선정한다.

- **설명 가능성:** 신입 직원에게 5일 안에 가르칠 수 있는 업무인가?
- **텍스트 기반 처리:** 입력 정보 10건을 모으면 결과물을 만들어낼 수 있는가?
- **빠른 검증:** 결과의 품질을 30분 안에 판단할 수 있는가?
- **정량 측정:** 처리 시간, 정확도 등을 수치로 측정할 수 있는가?

이 네 가지를 통과한 업무에 대해, 추가로 암묵지 비율(경험 없이도 가능한가), 업무 경로의 예측 가능성(예외 상황이 많은가), 관련 인원 수(완료까지 몇 명과 소통해야 하는가)를 평가한다. 이 결과가 이후 Layer 분류의 근거가 된다.

이 판단 기준은 곧 B2B 컨설팅의 첫 번째 진단 도구가 된다.

#### Step 2. 병목 진단 — "어떤 업무부터 해결해야 ROI가 가장 큰가?"

부서별 인터뷰를 통해 생산성 병목을 파악하고, 업무별 현재 평균 처리 시간을 측정해 기준선을 확보한다. 이 기준선이 이후 모든 성과 측정의 출발점이 된다. 시작점 선정 기준은 명확하다:

> **규칙이 보이고, 평가가 빠르고, 측정이 되고, 사람 의존성이 과도하지 않은 업무.**

이 기준에 부합하는 업무가 Layer 1이고, 곧 첫 번째 AI 에이전트의 대상이 된다. 여기서 수집한 인터뷰 결과와 병목 분석 데이터는 B2B 워크숍과 컨설팅의 실사례 콘텐츠가 된다.

#### Step 3. 3-Layer 분류 — "도입 순서를 어떻게 잡을 것인가?"

인터뷰 결과를 뤼튼의 부서별 업무 분류표와 대조한다. 뤼튼이 이미 8개 부서x3개 계층으로 분류한 업무 중 우리와 겹치는 것은 그대로 차용하고, 우리 고유의 업무만 자체 분류한다. 분류 기준은 단순하다:

- 의사결정이 필요 없으면 → Layer 1
- 데이터 분석은 필요하지만 최종 판단은 아니면 → Layer 2
- 최종 의사결정이나 외부 시스템 실행이 필요하면 → Layer 3

이 분류표는 우리 회사의 AX 로드맵인 동시에, B2B 컨설팅에서 고객사에 적용할 방법론 템플릿이 된다. 이 단계에서 병렬 트랙 B(기술력 진단)의 결과가 합류하여, 기술적으로 실행 가능한 현실적 로드맵이 완성된다.

#### Step 4. 구축과 측정 — "Layer 1부터 만들고, 성과를 재고, 다음으로 간다"

Layer 1을 먼저 구축해서 빠르게 성과를 확인한다. **Layer 1의 성과 데이터가 곧 Layer 2 투자의 승인 근거**다. 뤼튼의 경우 고객 응대 Layer 1(FAQ 자동 응답)만으로 건당 처리 시간이 19분에서 3분으로 줄었고, 이 숫자가 나머지 계층 투자를 정당화했다.

구축과 동시에 도입 전후 성과를 측정하고, 트러블슈팅 과정을 문서화한다. Layer 1이 성공적으로 안착하면 Layer 2로 확장하고, 같은 사이클을 반복한다. 이 전체 과정을 자사 AX 성과 보고서로 정리하면, 그것이 곧 B2B 패키지 영업의 핵심 제안서가 된다.

### 2-3. 성공 요소 2 — 기술력: 필수 기술 체크리스트와 고도화 전략

병목 진단(요소 1)과 병렬로, Agent 플랫폼 제공에 필요한 기술 역량을 점검하고 고도화 계획을 수립한다.

#### 필수 기술 체크리스트

Agent 플랫폼 사업에 필요한 핵심 기술 항목을 나열하고, 각 항목에 대해 현재 우리의 준비 수준을 평가한다. 이 체크리스트는 추가 리서치를 통해 구체화해야 하며, 아래는 뤼튼의 기술 스택과 시장 요구 사항에서 도출한 주요 항목이다.

- **RAG (Retrieval-Augmented Generation)** — 사내/고객사 문서 기반 응답 생성
- **Multi-Agent Architecture** — 복수 에이전트 간 협업 및 워크플로우 설계
- **Model Orchestration** — 작업 특성에 따른 최적 모델 자동 배치
- **sLLM Fine-tuning** — 도메인 특화 소형 모델 학습
- **Tool Calling / MCP** — 외부 시스템(API, DB 등)과의 연동
- **Vector DB & Semantic Search** — 유사도 기반 정보 검색
- **모델 평가 방법론** — 모델별 강점 프로파일링 및 품질 측정 (KLAUD 참고)

각 항목에 대해 현재 수준(미보유/기초/중급/고급)을 평가하고, 항목별 고도화 세부 전략을 수립한다. 이 과정에서 자체 구축할 기술과 외부 도구로 해결할 기술의 경계를 명확히 설정한다.

#### 모델 스펙 정리표

고객사에 맞는 모델을 추천하는 능력 자체가 핵심 기술 역량이다. 외부 API 모델(Claude, GPT 등)과 로컬 모델(오픈소스 sLLM 등)을 분리하여, 각각의 성능/비용/보안/속도 스펙을 정리한 표를 구축한다. 이 표는 실패 요소 Quality와 Price 대응의 기반 도구가 된다.

#### Dogfooding과 B2B에서의 역할

기술력은 Dogfooding 과정에서 실증적으로 검증되고 성장한다. 자사에 적용하면서 각 기술의 실제 성능과 한계를 확인하고, 그 경험이 B2B 구축 단계의 실행력으로 직결된다. 또한 모델 평가 방법론은 성과 측정 체계의 일부이자, "왜 이 모델을 이 업무에 배치했는가"를 설명하는 컨설팅 상품이 된다.

### 2-4. 성공 요소 3 — 경영진의 의지와 이해: 성과 측정 체계

경영진을 설득하고 지속적 투자를 이끌어내려면, AI 도입의 효과를 "체감상 좋아졌다"가 아닌 정량적 수치로 제시할 수 있어야 한다. 이 수치는 내부적으로는 경영진 설득의 근거, 외부적으로는 B2B 영업의 핵심 자산이 된다.

**투입 공수 측정** — 업무별 평균 처리 시간을 AI 도입 전후로 비교한다. "고객 응대 부서의 월간 투입 공수가 1,543분에서 423분으로 73% 감소"와 같은 구체적 숫자를 산출한다. 별도의 시스템 없이 지금 바로 측정을 시작할 수 있다.

**생산성 지수** — 뤼튼의 AI Productivity Index를 차용해 "인간+AI 협업 생산성"을 단일 숫자로 표현한다. 공식의 정교함보다 **"수학적 모델로 생산성을 측정합니다"라는 사실 자체**가 고객에게 신뢰를 준다. 뤼튼도 1차 적용에서는 단순화 버전으로 시작했다.

**모델 평가 방법론 (중장기)** — 뤼튼의 KLAUD를 참고해 우리 도메인에 맞는 모델 평가 기준을 만든다. "왜 이 업무에 이 모델을 배치했는가"를 설명할 수 있으면, 그 자체가 컨설팅 상품이 된다.

Layer 1 Quick Win 전략과 결합하면: 빠르게 성과 수치를 만들고 → 경영진에게 보고하고 → 다음 Layer 투자를 승인받는 사이클이 돌아간다.

### 2-5. 실패 5요소 가드레일

#### Quality — 모델 선택 역량

외부 API 모델(Claude, GPT 등)과 로컬 모델(오픈소스 sLLM 등)을 분리하여 각각의 성능/비용/보안/속도 스펙을 정리한 모델 매핑표를 구축한다. 고객사의 보안 요구, 성능 요구, 환경 제약에 맞는 최적 모델을 추천하는 것 자체가 컨설팅의 기술 역량이다. 뤼튼의 KLAUD처럼 모델별 강점 영역을 프로파일링하되, 우리 도메인에 맞는 평가 기준을 설계한다.

#### Price — 비용 제약 안에서 최대 성과를 내는 설계

고객사의 비용 제한은 환경 제약의 가장 큰 변수다. 프로젝트 시작 시 비용 제약을 먼저 파악하고, Quality에서 구축한 모델 매핑표를 기반으로 비용 범위 안에서 최적 조합을 설계한다. 성과 측정 체계는 "이 비용을 써서 이만큼 줄었다"는 비용 정당화의 근거로 활용한다. 비용 자체를 줄이기 위한 Model Orchestration(작업별 모델 분배), Caching 전략(반복 요청 재활용) 등의 기술도 연구한다.

#### Data Infra — 자사 데이터 정리 경험을 컨설팅 상품으로

우리 회사의 데이터 환경도 정리가 필요한 상태다. 이를 역으로 활용한다. 자사 데이터를 정리하는 과정 자체에서 "어떤 순서로, 어떤 기준으로, 어떤 도구로 정리해야 하는가"의 가이드라인과 노하우가 축적되고, 이것이 데이터 환경이 미비한 고객사를 대상으로 한 컨설팅 상품이 된다. Dogfooding의 논리와 동일하다.

#### UX — 선택지 제공 + 피드백 반영 체계

기존 도구(Slack 등)를 인터페이스로 쓸지, 새로운 전용 UX를 제공할지 고객사에 선택지를 준다. 양쪽 모두 제공할 수 있도록 제품을 구현해둔다. 이에 더해, 사용자 요구사항을 좋은 UX로 변환하는 설계 노하우와 기존 사용자 불만을 해결한 케이스를 체계적으로 축적한다. A/B 테스트 체계를 갖추고, 사용자 피드백을 빠르게 수집하고 반영할 수 있는 프로세스를 만들어, "만들었는데 안 쓰는" 상황을 구조적으로 방지한다.

#### AI Literacy — 레벨별 교육 프로그램 + CX Agent 확장

일회성 워크숍이 아니라, 개발자/비개발자 등 역할과 수준에 따른 레벨별 교육 프로그램을 설계한다. 우리 회사가 개발자/비개발자 혼합 구조이므로, 사내에서 먼저 체계를 잡으면 고객사에도 그대로 적용할 수 있다.

교육의 자연스러운 확장은 **CX(사용 중 문제 대응)**다. 교육을 받아도 실무에서 질문은 계속 생긴다. 이를 상시 지원하는 CX Agent를 구축한다:

- **제품 CX** — 우리 Agent 제품 사용 중 발생하는 문제에 대응 (매뉴얼/트러블슈팅 문서 기반 RAG)
- **AI 활용 CX** — "이 업무에 AI를 어떻게 쓰지?" 같은 범용 질문 대응 (교육 콘텐츠 + 사내 유즈케이스 기반)

CX Agent로 들어오는 질문 패턴은 다시 교육 프로그램 개선의 피드백이 되어 순환한다. 또한, 이 CX Agent 자체가 사내 Agent 사업의 하나로 구성될 수 있다.

### 2-6. 횡단 전략: Dogfooding

위의 모든 요소에 걸쳐 작동하는 전략이 Dogfooding이다. 우리가 만든 AI 에이전트를 우리 업무에 먼저 적용하는 것이 전략의 출발점이며, 뤼튼의 AX Report 전체가 **"우리가 직접 써봤고, 이만큼 줄었다"**를 말하기 위한 문서다.

Dogfooding은 단순히 "내부 테스트"가 아니라, 제품과 사업 역량을 동시에 성장시키는 체계다. 세 가지 자산이 축적된다:

**영업 신뢰 자산** — "이론적으로 좋습니다"가 아니라 "우리가 적용해보니 73% 줄었습니다." 자사 성과 보고서가 곧 B2B 제안서의 핵심이 된다.

**트러블슈팅 노하우** — AI 오답 처리, 기존 시스템 연동, 데이터 품질 문제, 직원의 사용 저항, 응답 속도 최적화 등은 직접 구축하고 운영해봐야만 얻는 경험이다. 이 경험이 컨설팅의 핵심 가치이자, 기술이 상향평준화된 시장에서 유일한 차별화 요소다.

**제품 개선 루프** — 자사 적용 → 문제 발견 → 개선 → 재적용 → 고객사 제공. 이 루프를 빠르게 돌릴수록 제품 품질이 올라간다.

| 자산 유형 | 예시 | B2B 활용 |
|---------|------|---------|
| **트러블슈팅 케이스** | AI 오답 발생 시 신뢰도 임계값 조정, 오류 시 대체 경로 설계 | 컨설팅 시 "이런 문제는 이렇게 해결합니다" |
| **UX 실패/성공 사례** | 구성원이 도구를 사용하지 않는 문제를 어떻게 극복했는가 | 도입 저항을 줄이는 UX 설계 노하우 |
| **비용 최적화 노하우** | 작업별 최적 모델 배치로 비용을 절감한 사례 | 비현실적 기대에 대한 현실적 대안 제시 |
| **도입 저항 관리** | Layer 1부터 점진적 도입으로 저항을 최소화한 과정 | 조직 변화관리 컨설팅 역량 |
| **데이터 품질 대응** | 사내 데이터가 정리되지 않은 상태에서 어떻게 시작했는가 | 데이터 환경이 미비한 기업 대상 솔루션 |

이 자산들은 반복할수록 정교해지는 하나의 라이프사이클로 성장한다:

```
[1] 자사 업무에 에이전트 적용
         ↓
[2] 문제 발생 → 트러블슈팅 → 해결책 문서화
         ↓
[3] 해결책을 제품에 반영 → 에이전트 품질 향상
         ↓
[4] 향상된 제품을 다시 자사에 적용 → 새로운 업무 영역으로 확장
         ↓
[5] 축적된 경험과 성과 데이터를 B2B 패키지에 반영
         ↓
[6] 고객사 적용 → 새로운 유즈케이스와 트러블슈팅 경험 추가 축적
         ↓
       (1로 돌아감)
```

이 사이클을 반복할수록 제품의 완성도, 트러블슈팅 역량, 성과 데이터가 동시에 두꺼워진다. 이 루프의 회전 속도와 축적량이 곧 우리만의 경쟁력이 된다.

### 2-7. 횡단 전략: 패키지형 B2B 판매

뤼튼의 고객 사례가 보여주듯, 도구만 납품하면 안 쓰이고(H사), 진단 없이 구축하면 방향이 틀리고(S사), 구축 역량 없이 컨설팅만 하면 실행이 안 된다(A사). 세 가지를 묶어야 한다.

| 패키지 | 내용 | Dogfooding에서 오는 것 |
|--------|------|----------------------|
| **교육 프로그램** | 레벨별 AI 이해도 교육 + 경영진 목표 정렬 | 도입 저항 극복 사례, 레벨별 커리큘럼 설계 경험 |
| **컨설팅** | 적합성 판단 → 인터뷰 → 3-Layer 분류 → 로드맵, 데이터 환경 진단, 모델 추천 | 필터링 기준, 분류 템플릿, 모델 매핑표, 데이터 정리 가이드라인 |
| **구축** | Layer 1 시범 → 성과 검증 → Layer 2~3 확장 | 트러블슈팅 케이스, 비용 최적화 노하우, 구축 속도 |
| **CX (상시 지원)** | 제품 사용 문제 대응 + AI 활용 범용 질문 대응 | CX Agent 구축 경험, 질문 패턴 기반 교육 개선 루프 |

**수익은 단계별로 확장된다:**

```
교육(진입) → 컨설팅(진단) → Layer 1 구축(성과 입증) → Layer 2~3 확장(업셀) → CX + 유지보수(정기 수익)
```

## 참고. 뤼튼 실증 데이터에서 읽는 영역별 AX 난이도

향후 에이전트 플랫폼(뤼튼에서는 "Compound Agent Ecosystem"이라 부르는, 여러 업무 영역의 에이전트를 하나의 생태계로 묶어 제공하는 형태)을 B2B로 판매한다면, 어떤 영역부터 진입하는 것이 유리한지 뤼튼의 실증 데이터가 참고가 된다.

**고객 응대(CX)가 가장 쉽다.** 4개 영역 중 투입 공수 감소율 1위(73%), 생산성 증가율 1위(35%)다. 업종과 규모에 상관없이 모든 기업에 존재하는 업무이고, Layer 1(FAQ 자동 응답)만으로 건당 처리 시간 85% 감소라는 즉각적 성과가 나온다. 뤼튼도 이 영역을 Phase 2에서 100% 자동화 목표로 가장 먼저 완성하고 있으며, 실제로 "상담사 0명 AICC 구축" 같은 구체적 수요가 고객사에서 들어오고 있다.

**개발 영역에서는 백엔드가 프론트엔드보다 쉽다.** 백엔드(55% 공수 감소)가 프론트엔드(28%)를 크게 앞선다. API 명세와 DB 스키마가 정형화되어 있어 AI가 처리하기 쉽고, 테스트 코드로 결과를 자동 검증할 수 있기 때문이다. 도입 후 분석/개선 자동화율에서도 백엔드(52%)가 프론트엔드(13%)를 압도한다. 프론트엔드는 "보기 좋아야 한다"는 비정형 기준이 존재하고, 시각적 검증이 필요해 자동화 난이도가 높다.

**CI/CD(빌드/배포 자동화)는 개발 AX의 가장 쉬운 진입점이다.** 프론트엔드(85%)와 백엔드(86%) 모두에서 가장 높은 자동화율을 기록했다. 반면, 요구사항 정의("무엇을 만들지")는 자동화율 0~6%로, 여전히 사람의 판단이 필요한 영역이다.
