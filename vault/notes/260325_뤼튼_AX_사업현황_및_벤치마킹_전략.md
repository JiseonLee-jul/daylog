# 뤼튼의 AX 사업 현황 및 우리의 벤치마킹 전략

*2025.1H K-AI Leadership Summit 발표 자료 기반 정리*

## Part 1. 뤼튼의 AX 사업 현황

### 1-1. 뤼튼은 누구인가

뤼튼테크놀로지스는 "Bring AGI Close to People"을 비전으로, 1년 7개월 만에 MAU 700만을 달성한 국내 1위 AI 서비스 기업이다. 2025년 3월 Series B 830억원을 유치했으며, 2024년 9월 B2B 전담 조직인 **wrtn.ax**를 설립해 삼성, CJ, KT, LG, 기아 등 대기업과 정부기관을 대상으로 AI 전환(AX) 사업을 본격화하고 있다.

### 1-2. AX에 대한 뤼튼의 관점

> "AX는 에이전트를 구축하는 프로젝트가 아니라, 생산성 병목을 해결하는 프로젝트다."

뤼튼은 AX를 기술 도입이 아닌 **경영 혁신**으로 정의한다. 기술은 이미 상향평준화되었고, 차이를 만드는 것은 "어떤 병목을 찾아 어떻게 해결했는가"의 실전 경험이라는 것이 핵심 메시지다.

### 1-3. 자사 AX 실증 결과

뤼튼은 자사 4개 부서에 AI 에이전트를 직접 적용하고, 그 성과를 수치로 공개했다.

| 적용 영역 | 역할 | 생산성 증가 | 월간 투입 공수 감소 |
|---------|------|---------|--------------|
| **고객 응대 (CX)** | 고객 문의 응답부터 환불/재배송 처리까지 자동화 | 35% | 73% (1,543분 → 423분) |
| **재무 (Finance)** | 회계, 지출, 급여, 재무제표 작성 자동화 | 21% | 40% (823분 → 496분) |
| **프론트엔드 개발** | 디자인 → 코드 변환, API 연동, 빌드/테스트 자동화 | 15% | 28% (3,500분 → 2,513분) |
| **백엔드 개발** | 자연어 → API/DB 설계, 테스트, 코드 자동 생성 | 28% | 55% (3,000분 → 1,348분) |

### 1-4. 핵심 방법론

#### 3-Layer 프레임워크

뤼튼은 모든 업무를 난이도에 따라 3개 계층으로 분류하고, 아래 계층부터 순차적으로 자동화한다.

| 계층 | 업무 성격 | 예시 |
|------|---------|------|
| **Layer 1** | 반복적이고 규칙이 명확한 업무 | FAQ 응답, 급여 명세서 처리, 빌드 스크립트 작성 |
| **Layer 2** | 데이터 분석과 일차적 판단이 필요한 업무 | 고객별 맞춤 응답, 재무 분석 리포트, 요구사항 분석 |
| **Layer 3** | 최종 의사결정과 외부 시스템 연동이 필요한 업무 | 환불 실행, 재무 예측(FP&A), 코드 품질 분석 후 개선 |

이 프레임워크의 본질은 **도입 순서**다. Layer 1에서 빠르게 성과를 내고, 그 수치를 근거로 다음 계층의 투자를 정당화하는 구조다.

#### 성과 측정 체계

뤼튼은 두 가지 지표로 AX의 효과를 정량화한다.

**투입 공수 측정:** 모든 업무를 최소 단위(건)로 정의하고, 건당 평균 처리 시간을 AI 도입 전후로 비교한다. 예를 들어 고객 응대 부서의 FAQ 처리는 건당 19분에서 3분으로 줄었다(-85%).

**생산성 지수:** 자체 개발한 수학적 모델(AI Productivity Index)로 "인간+AI 협업 시 총 생산성이 얼마나 증가했는가"를 단일 숫자로 산출한다. "35% 생산성 증가"처럼 경영진과 고객사에 직관적으로 전달할 수 있는 형태다.

### 1-5. B2B 사업 모델: AX Triangle Package

뤼튼은 제품 단독 판매가 아닌, 세 가지를 묶은 패키지 형태로 B2B 사업을 전개한다.

| 구성 | 내용 |
|------|------|
| **AI 역량 강화 프로그램** | 고객사 구성원의 AI 이해도를 높여 도입 저항을 줄이는 교육 |
| **AX 컨설팅** | 기업별 상황에 맞는 병목 진단, AX 로드맵 수립, 에이전트 구조 설계 |
| **에이전트 구축** | 로드맵 우선순위에 따른 실제 AI 에이전트 개발 및 배포 |

뤼튼이 발견한 실패들은 공통적으로 **진단과 교육 없이 구축부터 시작**했거나, **구축 역량 없이 계획만 세운** 경우다.

## Part 2. 우리의 벤치마킹 전략

### 2-1. 전략의 핵심 구조

```
자사에 먼저 적용(Dogfooding) → 성과를 수치로 증명 → 패키지형 B2B 사업화
```

뤼튼이 검증한 이 흐름을 우리도 따른다. 자사 적용에서 나오는 트러블슈팅 경험이 컨설팅의 가치가 되고, 측정 수치가 영업의 무기가 되며, 고객사 프로젝트가 다시 경험을 축적시키는 선순환 구조다.

### 2-2. 도입 프로세스

**Step 1 → 2를 건너뛰고 3 → 4로 가면 실패한다.** 뤼튼이 관찰한 시장 실패의 공통 원인이다. 아래 순서를 지키는 것이 전략의 전제 조건이다.

#### Step 1. 적합성 판단 — "이 업무에 AI를 적용할 수 있는가?"

모든 업무에 AI를 적용할 수 있는 것은 아니다. 인터뷰로 업무를 수집한 뒤, 다음 네 가지 조건을 모두 충족하는 업무만 AX 대상으로 선정한다.

- **설명 가능성:** 신입 직원에게 5일 안에 가르칠 수 있는 업무인가?
- **텍스트 기반 처리:** 입력 정보 10건을 모으면 결과물을 만들어낼 수 있는가?
- **빠른 검증:** 결과의 품질을 30분 안에 판단할 수 있는가?
- **정량 측정:** 처리 시간, 정확도 등을 수치로 측정할 수 있는가?

이 네 가지를 통과한 업무에 대해, 추가로 암묵지 비율(경험 없이도 가능한가), 업무 경로의 예측 가능성(예외 상황이 많은가), 관련 인원 수(완료까지 몇 명과 소통해야 하는가)를 평가한다. 이 결과가 이후 Layer 분류의 근거가 된다.

이 판단 기준은 곧 B2B 컨설팅의 첫 번째 진단 도구가 된다. 고객사를 만났을 때 "어떤 업무부터 AI를 적용할 수 있는가"를 체계적으로 판단하는 프레임워크로 그대로 활용할 수 있다.

#### Step 2. 병목 진단 — "어떤 업무부터 해결해야 ROI가 가장 큰가?"

부서별 인터뷰를 통해 생산성 병목을 파악하고, 업무별 현재 평균 처리 시간을 측정해 기준선(Baseline)을 확보한다. 이 기준선이 이후 모든 성과 측정의 출발점이 된다. 시작점 선정 기준은 명확하다:

> **규칙이 보이고, 평가가 빠르고, 측정이 되고, 사람 의존성이 과도하지 않은 업무.**

이 기준에 부합하는 업무가 Layer 1이고, 곧 첫 번째 AI 에이전트의 대상이 된다. 여기서 수집한 부서별 인터뷰 결과와 병목 분석 데이터는 B2B 워크숍과 컨설팅에서 "우리 회사에서는 이런 과정을 거쳐 이런 병목을 발견했고, 이렇게 우선순위를 정했다"는 실사례로 직접 활용할 수 있다.

#### Step 3. 3-Layer 분류 — "도입 순서를 어떻게 잡을 것인가?"

인터뷰 결과를 뤼튼의 부서별 업무 분류표와 대조한다. 뤼튼이 이미 8개 부서x3개 계층으로 분류한 업무 중 우리와 겹치는 것은 그대로 차용하고, 우리 고유의 업무만 자체 분류한다. 분류 기준은 단순하다:

- 의사결정이 필요 없으면 → Layer 1
- 데이터 분석은 필요하지만 최종 판단은 아니면 → Layer 2
- 최종 의사결정이나 외부 시스템 실행이 필요하면 → Layer 3

이렇게 완성된 3-Layer 분류표는 우리 회사의 AX 로드맵인 동시에, B2B 컨설팅에서 고객사에 동일한 분류 작업을 수행할 때의 방법론 템플릿이 된다.

#### Step 4. 구축과 측정 — "Layer 1부터 만들고, 성과를 재고, 다음으로 간다"

Layer 1을 먼저 구축해서 빠르게 성과를 확인한다. **Layer 1의 성과 데이터가 곧 Layer 2 투자의 승인 근거**다. 뤼튼의 경우 고객 응대 Layer 1(FAQ 자동 응답)만으로 건당 처리 시간이 19분에서 3분으로 줄었고, 이 숫자가 나머지 계층 투자를 정당화했다.

구축과 동시에 도입 전후 성과를 측정하고, 트러블슈팅 과정을 문서화한다. Layer 1이 성공적으로 안착하면 Layer 2로 확장하고, 같은 사이클을 반복한다. 이 전체 과정을 자사 AX 성과 보고서로 정리하면, 그것이 곧 B2B 패키지 영업의 핵심 제안서가 된다.

### 2-3. 성과 측정 체계

AI 도입의 효과를 "체감상 좋아졌다"가 아닌 정량적 수치로 제시할 수 있어야 한다. 이 수치가 내부적으로는 경영진 설득의 근거, 외부적으로는 B2B 영업의 핵심 자산이 된다.

**투입 공수 측정** — 업무별 평균 처리 시간을 AI 도입 전후로 비교한다. 뤼튼의 사례처럼 "고객 응대 부서의 월간 투입 공수가 1,543분에서 423분으로 73% 감소"와 같은 구체적 숫자를 산출한다. 별도의 시스템 없이 지금 바로 측정을 시작할 수 있다.

**생산성 지수** — 뤼튼의 AI Productivity Index를 차용해 "인간+AI 협업 생산성"을 단일 숫자로 표현한다. 공식의 정교함보다 **"수학적 모델로 생산성을 측정합니다"라는 사실 자체**가 고객에게 신뢰를 준다. 뤼튼도 1차 적용에서는 단순화 버전으로 시작했다.

**AI 모델 평가 방법론 (중장기)** — 뤼튼의 KLAUD(한국어 LLM 평가 방법론)를 참고해 우리 도메인에 맞는 모델 평가 기준을 만든다. "왜 이 업무에 이 모델을 배치했는가"를 설명할 수 있으면, 그 자체가 컨설팅 상품이 된다.

### 2-4. Dogfooding: 자사 적용이 만드는 자산과 체계

우리가 만든 AI 에이전트를 우리 업무에 먼저 적용하는 것이 전략의 출발점이다. 뤼튼의 AX Report 전체가 "우리가 직접 써봤고, 이만큼 줄었다"를 말하기 위한 문서다. Dogfooding은 단순히 "내부 테스트"가 아니라, 제품과 사업 역량을 동시에 성장시키는 체계다.

#### 축적되는 세 가지 자산

**영업 신뢰 자산** — "이론적으로 좋습니다"가 아니라 "우리가 적용해보니 73% 줄었습니다." 자사 성과 보고서가 곧 B2B 제안서의 핵심이 된다.

**트러블슈팅 노하우** — AI 오답 처리, 기존 시스템 연동, 데이터 품질 문제, 직원의 사용 저항, 응답 속도 최적화 등은 직접 구축하고 운영해봐야만 얻는 경험이다. 이 경험이 컨설팅의 핵심 가치이자, 기술이 상향평준화된 시장에서 유일한 차별화 요소다.

**제품 개선 루프** — 자사 적용 → 문제 발견 → 개선 → 재적용 → 고객사 제공. 이 루프를 빠르게 돌릴수록 제품 품질이 올라간다.

| 자산 유형 | 예시 | B2B 활용 |
|---------|------|---------|
| **트러블슈팅 케이스** | AI 오답(Hallucination) 발생 시 신뢰도 임계값 조정, 오류 시 대체 경로 설계 경험 | 컨설팅 시 "이런 문제는 이렇게 해결합니다" |
| **UX 실패/성공 사례** | 구성원이 도구를 사용하지 않는 문제를 어떻게 극복했는가 | 도입 저항을 줄이는 UX 설계 노하우 |
| **비용 최적화 노하우** | 작업별 최적 모델 배치로 비용을 절감한 사례 | 비현실적 기대("최저 비용으로 최고 성능")에 대한 현실적 대안 제시 |
| **도입 저항 관리** | Layer 1부터 점진적 도입으로 저항을 최소화한 과정 | 조직 변화관리 컨설팅 역량 |
| **데이터 품질 대응** | 사내 데이터가 정리되지 않은 상태에서 어떻게 시작했는가 | 데이터 환경이 미비한 기업 대상 솔루션 |

#### Dogfooding이 만드는 제품 라이프사이클

이 자산들은 개별적으로 쌓이는 것이 아니라, 반복할수록 정교해지는 하나의 체계로 성장한다.

```
[1] 자사 업무에 에이전트 적용
         ↓
[2] 문제 발생 → 트러블슈팅 → 해결책 문서화
         ↓
[3] 해결책을 제품에 반영 → 에이전트 품질 향상
         ↓
[4] 향상된 제품을 다시 자사에 적용 → 새로운 업무 영역으로 확장
         ↓
[5] 축적된 경험과 성과 데이터를 B2B 패키지에 반영
         ↓
[6] 고객사 적용 → 새로운 유즈케이스와 트러블슈팅 경험 추가 축적
         ↓
       (1로 돌아감)
```

이 사이클을 반복할수록 제품의 완성도, 트러블슈팅 역량, 성과 데이터가 동시에 두꺼워진다. 뤼튼이 Phase 1에서 Phase 2로 진화하는 과정이 정확히 이 루프이며, 이 루프의 회전 속도와 축적량이 곧 우리만의 경쟁력이 된다.

### 2-5. B2B 사업 모델: 패키지형 판매

뤼튼의 고객 사례가 보여주듯, 도구만 납품하면 안 쓰이고(H사), 진단 없이 구축하면 방향이 틀리고(S사), 구축 역량 없이 컨설팅만 하면 실행이 안 된다(A사). 세 가지를 묶어야 한다.

| 구성 | 내용 | 해결하는 문제 |
|------|------|------------|
| **AX Readiness 워크숍** | AI 이해도 교육 + 경영진 목표 정렬 | "어디서 시작할지 모르겠다" / 구성원 사용 저항 |
| **병목 중심 컨설팅** | 적합성 판단 → 부서 인터뷰 → 3-Layer 분류 → 로드맵 수립, 데이터 환경 진단 | "비현실적 로드맵" / 잘못된 우선순위 |
| **계층별 단계 구축** | Layer 1 시범 적용 → 성과 검증 → Layer 2~3 확장 | "만들었는데 안 쓴다" / "제대로 하는 곳이 없다" |

**수익은 단계별로 확장된다:**

```
워크숍(진입) → 컨설팅(진단) → Layer 1 구축(성과 입증) → Layer 2~3 확장(업셀) → 유지보수(정기 수익)
```

#### Dogfooding 경험이 패키지 각 단계에 기여하는 방식

**워크숍** — Dogfooding 과정에서 직접 겪은 도입 저항 사례(구성원이 안 쓰는 문제, UX 개선 과정 등)가 교육 콘텐츠가 된다. "일반론"이 아니라 "우리가 겪은 사례"로 설명할 수 있어 설득력이 다르다. 또한 사내 데이터 환경 진단 경험이 고객사 데이터 환경을 평가하는 프레임워크로 바로 전환된다.

**컨설팅** — Step 1~3에서 축적한 적합성 판단 기준, 인터뷰 방법론, 3-Layer 분류 템플릿을 고객사에 그대로 적용할 수 있다. 특히 "이 업무는 AI에 적합하지 않다"고 판단하는 근거를 자사 실험 데이터로 제시할 수 있으면, 과잉 기대를 사전에 조율하고 컨설팅의 신뢰도를 높인다.

**구축** — 트러블슈팅 케이스와 비용 최적화 노하우가 구축 품질을 직접 좌우한다. 자사에서 이미 해결한 문제를 고객사에서 반복하지 않으므로 구축 속도가 빨라지고, "이런 상황에서는 이렇게 대응했다"는 레퍼런스가 고객사의 신뢰를 만든다.

#### AX 실패 5요소와 우리의 회피 전략

뤼튼은 AX가 실패하는 원인을 5가지로 정리했다. 각 요소에 대해 뤼튼의 대응 방식과 우리가 취할 전략을 함께 정리한다.

| 실패 요소 | 뤼튼의 대응 | 우리의 대응 전략 |
|---------|---------|------------|
| **품질 (Quality)** | 자체 평가 방법론(KLAUD)으로 모델별 강점을 파악하고, 작업별 최적 모델을 배치 | 모델 평가 방법론을 차용하거나 간소화 버전 구축. 비현실적 성능 기대는 사전에 관리 |
| **비용 (Price)** | 저비용 ML 모델과 고비용 LLM을 하이브리드로 결합해 비용 통제 | 규칙 기반으로 해결 가능한 작업은 LLM을 쓰지 않음. 비용 구조를 투명하게 공개 |
| **데이터 환경 (Data Infra)** | 사내 데이터의 디지털화를 AX의 선행 조건으로 설정 | 컨설팅 단계에서 데이터 환경 수준을 진단하고, 미비하면 데이터 정비부터 시작 |
| **사용자 경험 (UX)** | AX 5대 원칙 중 하나로 강조. H사 실패 사례를 직접 경험 | Slack 등 기존에 쓰는 도구를 인터페이스로 활용. 새로운 도구 사용을 강제하지 않음 |
| **AI 이해도 (AI Literacy)** | B2B 패키지에 "AI 역량 강화 프로그램"을 필수로 포함 | 교육을 제품에 번들링. 도구만 납품하면 안 쓴다는 전제 하에 설계 |

## 참고. 뤼튼 실증 데이터에서 읽는 영역별 AX 난이도

향후 에이전트 플랫폼(뤼튼에서는 "Compound Agent Ecosystem"이라 부르는, 여러 업무 영역의 에이전트를 하나의 생태계로 묶어 제공하는 형태)을 B2B로 판매한다면, 어떤 영역부터 진입하는 것이 유리한지 뤼튼의 실증 데이터가 참고가 된다.

**고객 응대(CX)가 가장 쉽다.** 4개 영역 중 투입 공수 감소율 1위(73%), 생산성 증가율 1위(35%)다. 업종과 규모에 상관없이 모든 기업에 존재하는 업무이고, Layer 1(FAQ 자동 응답)만으로 건당 처리 시간 85% 감소라는 즉각적 성과가 나온다. 뤼튼도 이 영역을 Phase 2에서 100% 자동화 목표로 가장 먼저 완성하고 있으며, 실제로 "상담사 0명 AICC 구축" 같은 구체적 수요가 고객사에서 들어오고 있다.

**개발 영역에서는 백엔드가 프론트엔드보다 쉽다.** 백엔드(55% 공수 감소)가 프론트엔드(28%)를 크게 앞선다. API 명세와 DB 스키마가 정형화되어 있어 AI가 처리하기 쉽고, 테스트 코드로 결과를 자동 검증할 수 있기 때문이다. 도입 후 분석/개선 자동화율에서도 백엔드(52%)가 프론트엔드(13%)를 압도한다. 프론트엔드는 "보기 좋아야 한다"는 비정형 기준이 존재하고, 시각적 검증이 필요해 자동화 난이도가 높다.

**CI/CD(빌드/배포 자동화)는 개발 AX의 가장 쉬운 진입점이다.** 프론트엔드(85%)와 백엔드(86%) 모두에서 가장 높은 자동화율을 기록했다. 반면, 요구사항 정의("무엇을 만들지")는 자동화율 0~6%로, 여전히 사람의 판단이 필요한 영역이다.
